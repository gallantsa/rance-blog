### 从 Git 暂存区移除文件但不删除本地

项目初始化阶段如果无意添加了不必要的文件到 git 暂存区，可以使用以下命令将这些文件从暂存区移除，而不删除工作目录中的文件：

```bash
# 将.idea目录从git暂存区移除，但保留在本地
git rm --cached -r .idea/
```

## Django / Python 开发问题

### 1. `markdown.markdown()` 误用导致 `AttributeError`

**现象**：访问文章详情页时抛出 `AttributeError: 'str' object has no attribute 'convert'`

**根因**：`markdown.markdown()` 是快捷函数，返回 HTML 字符串，但代码将其当作 Markdown 实例调用了 `.convert()` 和 `.toc`。

```python
# ❌ 错误写法
md = markdown.markdown(extensions=[...])  # 返回 str（HTML）
post.body = md.convert(post.body)         # AttributeError：str 没有 .convert()
post.toc = md.toc                          # AttributeError：str 没有 .toc
```

**解决**：改用 `markdown.Markdown()` 类构造器，获得对象实例：

```python
# ✅ 正确写法
md = markdown.Markdown(extensions=[...])   # 返回 Markdown 实例
post.body = md.convert(post.body)          # ✅ 有 .convert() 方法
post.toc = md.toc                           # ✅ 有 .toc 属性（TocExtension注入）
```

| API | 返回类型 | 用途 |
|-----|---------|------|
| `markdown.markdown(text, ...)` | `str` | 快捷函数，一次性 Markdown → HTML |
| `markdown.Markdown(extensions=...)` | `Markdown` 实例 | 类构造器，需要获取 `.toc` 等附加属性时使用 |

### 2. `{% load %}` 缺失导致 `TemplateSyntaxError`

**现象**：访问博客页面时抛出 `TemplateSyntaxError: Invalid block tag on line: 'static'`

**根因**：重构侧边栏为自定义模板标签时，`base.html` 的 `{% load static %}` 被误替换为 `{% load blog_extras %}`，但模板头部仍在使用 `{% static '...' %}` 引用静态文件。

```html
<!-- ❌ 错误 — 只加载了自定义标签库 -->
{% load blog_extras %}

<!-- 以下 {% static %} 找不到标签库 → TemplateSyntaxError -->
<link rel="stylesheet" href="{% static 'blog/css/bootstrap.min.css' %}">
```

**解决**：同时加载两个标签库（先后顺序均可）：

```html
<!-- ✅ 正确 — 同时加载两个标签库 -->
{% load static %}
{% load blog_extras %}
```

**注意**：Django 模板中每个用到的标签库都需要 `{% load %}` 声明，加载多个可在多行中分别写，也可合并为 `{% load static blog_extras %}`。

### 3. 子模板中未加载自定义标签库 `TemplateSyntaxError`

**现象**：访问文章详情页时抛出 `Invalid block tag on line: 'show_comment_form'` 或类似错误。

**根因**：`templates/blog/detail.html` 中引入了评论模块的自定义模板标签 `{% show_comment_form %}` 和 `{% show_comments %}`（定义在 `comments/templatetags/comments_extras.py`），但模板顶部没有 `{% load comments_extras %}` 声明。

```html
<!-- ❌ 错误 — 没有加载 comments_extras 标签库 -->
{% extends 'base.html' %}

{% block main %}
  ...
  {% show_comment_form post %}    ← TemplateSyntaxError
  {% show_comments post %}        ← TemplateSyntaxError
  ...
{% endblock main %}
```

**解决**：在模板顶部加载对应的标签库：

```html
<!-- ✅ 正确 — 加载 comments_extras -->
{% extends 'base.html' %}
{% load comments_extras %}

{% block main %}
  ...
  {% show_comment_form post %}    ← ✅ 正常渲染
  {% show_comments post %}        ← ✅ 正常渲染
  ...
{% endblock main %}
```

**经验教训**：当 Django 模板中使用自定义标签（inclusion tag、simple tag 等）时，必须注意：
1. 确保标签库已创建（`templatetags/xxx.py` + `__init__.py`）
2. 确保标签库在 INSTALLED_APPS 中对应的 app 已注册
3. **在模板顶部显式 `{% load xxx %}` 声明**（继承的父模板的 load 不会传递给子模板）

### 4. Nginx 反向代理下 Django Admin CSRF 验证失败

**现象**：通过 Nginx 反向代理访问 Django Admin 后台，登录提交时提示 "CSRF verification failed"。

**根因**：Django 的 CSRF 中间件会检查请求的 `Origin` 或 `Referer` 头是否在 `CSRF_TRUSTED_ORIGINS` 白名单中。当经过 Nginx 反向代理时，Django 收到的请求来源与配置不匹配。

```python
# ❌ 缺少配置 — Django 拒绝所有非 localhost 的 POST 请求
# config/settings/production.py 中未设置 CSRF_TRUSTED_ORIGINS
```

**解决**：在 `production.py` 中添加两个配置：

```python
# ✅ 1. CSRF 信任所有来源（开发/测试环境可用）
CSRF_TRUSTED_ORIGINS = ['*']

# ✅ 2. 信任 Nginx 传递的 HTTPS 协议头
SECURE_PROXY_SSL_HEADER = ('HTTP_X_FORWARDED_PROTO', 'https')
```

**说明**：
- `CSRF_TRUSTED_ORIGINS = ['*']` — 允许所有来源（适合 Orbstack、localhost 等本地环境），生产环境建议限制为实际域名
- `SECURE_PROXY_SSL_HEADER` — 告诉 Django 信任 Nginx 传递的 `X-Forwarded-Proto: https` 头，确保重定向和链接使用 HTTPS
- 如果只允许特定域名，可以改为列表形式：`['https://your-domain.com']`

### 5. Python 3.14 中 `collections.Iterable` 被移除导致第三方库报错

**现象**：访问首页时抛出 `AttributeError: module 'collections' has no attribute 'Iterable'`。

**根因**：`django-pure-pagination` 库内部使用了 `collections.Iterable`，该接口在 Python 3.10 标记为废弃，Python 3.14 中正式移除。

```python
# ❌ 第三方库旧代码（pure_pagination/paginator.py）
elif isinstance(result, collections.Iterable):
    # ...
```

**解决**：在引入第三方库之前添加兼容性补丁：

```python
# ✅ 顶部添加（blog/views.py）
import collections
if not hasattr(collections, 'Iterable'):
    import collections.abc
    collections.Iterable = collections.abc.Iterable
```

**说明**：Python 3.14 中 `collections.abc.Iterable` 仍然可用，只需将 `collections.Iterable` 别名指向它即可。此补丁对 Python 低于 3.14 的版本无害（`hasattr` 检查为 True 时跳过）。

### 6. Django 5.x 中 `{% ifequal %}` 标签被移除导致 `TemplateSyntaxError`

**现象**：引入 `django-pure-pagination` 后访问首页抛出 `TemplateSyntaxError: Invalid block tag on line: 'ifequal'`。

**根因**：`django-pure-pagination` 自带的默认分页模板使用了 `{% ifequal %}` / `{% endifequal %}`，该标签在 Django 3.x 标记为废弃，Django 4.0 中正式移除。

```html
<!-- ❌ 第三方库旧模板 -->
{% ifequal page page_obj.number %}
  <li class="current"><a href="#">{{ page }}</a></li>
{% else %}
  <li><a href="?{{ page.querystring }}" class="page">{{ page }}</a></li>
{% endifequal %}
```

**解决**：在项目级模板目录中覆盖库的模板，将 `{% ifequal %}` 替换为 `{% if %}`：

```html
<!-- ✅ templates/pure_pagination/pagination.html -->
{% if page == page_obj.number %}
  <li class="current"><a href="#">{{ page }}</a></li>
{% else %}
  <li><a href="?{{ page.querystring }}" class="page">{{ page }}</a></li>
{% endif %}
```

**说明**：Django 模板查找机制会优先使用项目 `templates/` 目录中的模板，因此只需在 `templates/pure_pagination/pagination.html` 中提供修复版本即可，无需修改第三方库的原始文件。

### 7. URL 名称重构后模板中未同步更新导致 `NoReverseMatch`

**现象**：首页归档链接报错 `NoReverseMatch: Reverse for 'archive' not found`。

**根因**：类视图重构时将 URL name 从 `archive` 改为 `archives`（commit b17c514），但 `_archives.html` 模板中仍使用旧的 `{% url 'blog:archive' %}`。

```html
<!-- ❌ 类视图重构后，blog:archive 已不存在 -->
<a href="{% url 'blog:archive' date.year date.month %}">
```

**解决**：模板中同步更新为新的 URL name：

```html
<!-- ✅ 改为新名称 -->
<a href="{% url 'blog:archives' date.year date.month %}">
```

**经验教训**：重构 URL 路由时，务必同步搜索全项目的 `{% url %}` 模板标签和 `reverse()` 调用，确保无遗漏。

### 8. Elasticsearch 启动慢导致 Django 容器启动时连接失败

**现象**：生产环境 Django 容器启动后，搜索功能无结果，日志报 `ConnectionError: Failed to establish a new connection: Connection refused`。

**根因**：Elasticsearch 容器需要 10-30 秒完成启动，而 Django 容器的 `start.sh` 在 migrate 后立即尝试连接 ES，此时 ES 尚未就绪。

**解决**：在 `start.sh` 中加入 ES 就绪等待循环：

```bash
# ✅ 最多等待 60 秒（30 次 × 2 秒）
for i in $(seq 1 30); do
    if uv run python -c "
import os; os.environ['DJANGO_SETTINGS_MODULE']='config.settings.local'
import django; django.setup()
from haystack import connections
if connections['default'].get_backend().conn.ping():
    print('connected')
" 2>/dev/null | grep -q connected; then
        break
    fi
    sleep 2
done
```

### 9. `document_count()` 不存在导致索引策略判断失败

**现象**：智能索引脚本抛出 `AttributeError: 'Elasticsearch2IkSearchBackend' object has no attribute 'document_count'`。

**根因**：Haystack 的 `Elasticsearch7SearchBackend` 没有 `document_count()` 方法。

**解决**：在自定义后端中添加此方法：

```python
# ✅ blog/elasticsearch2_ik_backend.py
class Elasticsearch2IkSearchBackend(Elasticsearch7SearchBackend):

    def document_count(self):
        return self.conn.count(index=self.index_name)['count']
```

### 10. `rebuild_index` 内部 `clear_index` 的 `noinput` 参数透传失败

**现象**：`rebuild_index --noinput` 仍弹出确认提示，导致非交互环境报 `EOFError`。

**根因**：Haystack 的 `rebuild_index.handle()` 内部调用 `clear_index` 时没有正确传递 `noinput` 参数。

**解决**：改为两步走：

```python
# ❌ 不要使用 rebuild_index
call_command('rebuild_index', noinput=True)   # 内部 clear_index 会要求交互确认

# ✅ 改用 clear_index + update_index 两步
call_command('clear_index', noinput=True, interactive=False)
call_command('update_index', age=None)
```

### 11. `update_index` 不支持 `noinput` 参数导致 `TypeError`

**现象**：`call_command('update_index', noinput=True)` 报 `TypeError: Unknown option(s) for update_index command: noinput`。

**根因**：Haystack 的 `update_index` 命令没有定义 `--noinput` 选项（它不需要交互确认）。

**解决**：调用 `update_index` 时不传 `noinput`：

```python
# ❌ update_index 不支持 noinput
call_command('update_index', age=None, noinput=True)

# ✅ 只传 age 参数即可
call_command('update_index', age=None)
```

### 12. `assertQuerysetEqual` 在 Django 5.x 中已不存在

**现象**：运行测试时报 `AttributeError: 'TestCase' object has no attribute 'assertQuerysetEqual'`。

**根因**：Django 5.x 中将此方法重命名为 `assertQuerySetEqual`（注意大写 S）。

**解决**：

```python
# ❌ Django 5.x 前（已删除）
self.assertQuerysetEqual(qs, [repr(p) for p in qs])

# ✅ Django 5.x
self.assertQuerySetEqual(qs, [repr(p) for p in qs], transform=repr)
```
